# Пользовательские сценарии для проекта StudyBalance

---

## Сценарий 1 — Регистрация и вход в аккаунт

1. Пользователь открывает сайт StudyBalance в браузере.
2. На экране появляется форма регистрации: логин, пароль, электронная почта.
3. Пользователь вводит логин.
   - Если логин уже существует, система выводит сообщение:
     *«Такой логин уже используется. Пожалуйста, выберите другой.»*
4. Пользователь вводит пароль.
   - Если пароль короче 6 символов, система выводит:
     *«Пароль должен содержать от 6 до 30 символов.»*
5. Пользователь вводит адрес электронной почты.
   - Если формат неверный, система сообщает об ошибке.
6. Система отправляет письмо с подтверждением регистрации.
7. После подтверждения пользователь автоматически входит в систему.

---

## Сценарий 2 — Настройка учебного профиля

1. Пользователю предлагается выбрать учебный профиль (7–9 класс, 10–11 класс, колледж, другое).
2. При выборе пункта «другое» запускается сценарий создания кастомного профиля.
3. Система предлагает ознакомиться с особенностями выбранного профиля.
4. Пользователь подтверждает или пропускает ознакомление.
5. Далее открывается список предметов данного профиля.
6. Пользователь отмечает изучаемые предметы.
7. После сохранения запускается сценарий первого добавления задач.

---

## Сценарий 3 — Создание кастомного профиля

1. Пользователь видит список доступных функций профиля.
2. Напротив каждой функции размещён чекбокс.
3. Пользователь выбирает необходимые функции.
4. Система в реальном времени формирует итоговую конфигурацию профиля.
5. При нажатии «Далее» выводится предупреждение о временном ограничении изменений.
6. Пользователь подтверждает выбор или возвращается назад.
7. Профиль сохраняется и становится активным.

---

## Сценарий 4 — Первое добавление учебных задач

1. Пользователю отображается пустой список задач.
2. Пользователь нажимает кнопку «Добавить задачу».
3. Открывается форма ввода параметров задачи.
4. Пользователь заполняет название, предмет, тип, срок и сложность.
5. При ошибках система выводит предупреждающие сообщения.
6. После сохранения задача появляется в списке и календаре.
7. Приложение автоматически пересчитывает недельную нагрузку.

---

## Сценарий 5 — Просмотр недельного расписания

1. Пользователь открывает вкладку «Неделя».
2. Приложение отображает задачи по дням недели.
3. Загруженные дни подсвечиваются цветом.
4. Пользователь может редактировать или переносить задачи.
5. При переносе задач нагрузка пересчитывается автоматически.
6. При перегрузе система выводит предупреждение.
7. Изменения сразу сохраняются в базе данных.

---

## Сценарий 6 — Страница «Сегодня»

1. Пользователь открывает страницу «Сегодня».
2. Отображаются все задачи, запланированные на текущий день.
3. Приложение показывает общий уровень нагрузки.
4. Система формирует рекомендации по перераспределению задач.
5. Пользователь может отметить задачу выполненной.
6. Завершённые задачи перемещаются в архив.
7. Статистика обновляется автоматически.

---

## Сценарий 7 — Работа со статистикой

1. Пользователь переходит во вкладку «Статистика».
2. Приложение анализирует накопленные данные.
3. Отображаются графики нагрузки и выполнения задач.
4. Пользователь выбирает тип визуализации.
5. При наведении отображаются подробные данные.
6. Возможен экспорт графиков и таблиц.
7. Данные используются для построения рекомендаций.

---

## Сценарий 8 — Напоминания и уведомления

1. Пользователь открывает настройки уведомлений.
2. Выбирает типы напоминаний.
3. Устанавливает время получения уведомлений.
4. Система проверяет подтверждение электронной почты.
5. При необходимости запрашивается подтверждение email.
6. Напоминания сохраняются в профиле пользователя.
7. Система начинает отправку уведомлений автоматически.

---

## Сценарий 9 — Резервное копирование данных

1. Пользователь открывает раздел резервного копирования.
2. Выбирает тип сохранения данных.
3. Система информирует о составе резервной копии.
4. Пользователь подтверждает создание копии.
5. Формируется файл резервного сохранения.
6. Файл доступен для скачивания или хранения.
7. Пользователь может восстановить данные из копии.

---

## Сценарий 10 — Работа нескольких пользователей

1. Приложение развёрнуто на сервере.
2. Пользователи входят под своими логинами.
3. Каждый пользователь видит только свои данные.
4. Все операции изолированы по аккаунтам.
5. Одновременная работа пользователей поддерживается системой.
6. База данных корректно обрабатывает параллельные запросы.
7. Производительность сохраняется при нагрузке до 5 пользователей.
